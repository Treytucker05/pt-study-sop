# RQA-150 - Methods And Mastery Audit

## Scope

- `dashboard_rebuild/client/src/pages/methods.tsx`
- `dashboard_rebuild/client/src/pages/mastery.tsx`
- related components

## Findings

### RQA-150-001
- Severity: P2
- Type: ux
- Surface: `/methods` page and `MethodsPage` empty/loading states
- Summary: The methods lists (blocks/chains) show an empty-state message when their queries fail, because loading is only gated by `isLoading` and query errors are never handled explicitly. When the API rejects or network fails, users see “No methods found” / “No chains yet” as if data were simply absent, rather than a data error.
- Repro: 1) Make `api.methods.getAll` or `api.chains.getAll` return an HTTP/network error. 2) Navigate to `/methods`. 3) Observe that no explicit error alert appears; the UI shows empty-state text for zero items.
- Evidence:
  - `dashboard_rebuild/client/src/pages/methods.tsx` lines around list rendering for blocks: `if (blocksLoading) ... else ...` then `filteredBlocks.length === 0` empty message.
  - `dashboard_rebuild/client/src/pages/methods.tsx` lines around chain rendering: `chainsLoading ? Loading... : chains.length === 0 ? ...`.
  - `dashboard_rebuild/client/src/pages/methods.tsx` has no `isError` branch for either `useQuery` call; both queries default to `data: []` in render.
- Likely owner: Frontend (`dashboard_rebuild` route team for methods UI state handling)
- Fix wave: Wave 2

### RQA-150-002
- Severity: P2
- Type: ui
- Surface: `/mastery` diagnostics panel state
- Summary: The mastery diagnostics panel can remain visible after the selected skill is no longer in the returned skills list, because the selected skill ID is retained separately and used to open the why-locked panel regardless of whether that ID is still present in current `skills`.
- Repro: 1) Load `/mastery` with a locked skill selected. 2) Trigger a refetch where that skill disappears (for example, backend status transition/removal). 3) UI can still render `WhyLockedPanel` from stale `selectedSkillId` even when `selectedSkill` becomes `null`.
- Evidence:
  - `dashboard_rebuild/client/src/pages/mastery.tsx` state and selection logic: `const [selectedSkillId, setSelectedSkillId] = useState<string | null>(null);`
  - `const selectedSkill = skills.find((skill) => skill.skill_id === selectedSkillId) ?? null;`
  - Panel rendering condition uses `selectedSkillId && whyLocked` instead of checking `selectedSkill` as the source-of-truth.
  - Sidebar text path switches to “Select a locked skill...” when `selectedSkill` is null, while the right panel can still mount using stale id.
- Likely owner: Frontend (`dashboard_rebuild` mastery page/state coupling)
- Fix wave: Wave 2

## No findings beyond above

- No additional concrete /methods or /mastery runtime breakages were identified in the specified scope beyond the two items above.
