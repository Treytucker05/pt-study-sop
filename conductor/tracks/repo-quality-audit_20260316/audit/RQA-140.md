# RQA-140 - Scholar Audit

## Scope

- `dashboard_rebuild/client/src/pages/scholar.tsx`
- `dashboard_rebuild/client/src/components/ScholarRunStatus.tsx`
- related Scholar API route surfaces (`/api/scholar/status`, `/api/scholar/run`, `/api/scholar/run/status`)

## Findings

- ID: RQA-140-001
- Severity: P2
- Type: polling
- Surface: `/scholar` page (`scholar.tsx`) question polling
- Summary: `questionsQuery` polls `/api/scholar/research/questions?status=all&limit=100` every 2.5s whenever route is `/scholar`, regardless of whether an active investigation exists. This keeps backend question traffic hot during passive, idle Scholar viewing (or when no run is in progress), creating unnecessary load and noisy UI refreshes.
- Repro: Open `/scholar` with no selected/active investigation and keep the tab open for 10+ seconds. Network traffic shows recurring `scholar-research-questions` calls even though no user action or investigation activity is happening.
- Evidence: `dashboard_rebuild/client/src/pages/scholar.tsx:795-800` (`enabled: isScholarRoute`, fixed `refetchInterval: 2500`) and `dashboard_rebuild/client/src/pages/scholar.tsx:760-810` (neighboring queries are conditionally gated by run/investigation status).
- Likely owner: frontend
- Fix wave: Wave 2

- ID: RQA-140-002
- Severity: P2
- Type: contract
- Surface: Backend + API client status/run surfaces (`/api/scholar/status`, `api.scholar.runStatus`, `/api/scholar/run`, `/api/scholar/run/status`)
- Summary: Scholar run status and run-starting endpoints are split across two live surfaces (`api_adapter` and `dashboard_bp`) with non-aligned payload semantics. The frontend status card consumes `/api/scholar/status` via `api.scholar.runStatus`, while legacy dashboard routes still expose `/api/scholar/run` and `/api/scholar/run/status` with different status naming/shape expectations, increasing API contract drift risk.
- Repro: Compare endpoint wiring and handlers in the adapter and dashboard blueprint; no single schema is used for Scholar run intent/state (`running`/`complete` vs DB-style `success`/`failed`/`idle`) across routes.
- Evidence: `dashboard_rebuild/client/src/api.ts:530-568` (client route mappings for `run` and `runStatus`), `dashboard_rebuild/client/src/api.types.ts:315-327` (shared status type), `brain/dashboard/api_adapter.py:3567-3618` (`_scholar_status_payload` + `/scholar/status` handler at `:3641-3648`), `brain/dashboard/routes.py:1001-1039` (`/api/scholar/run`, `/api/scholar/run/status`), `brain/dashboard/app.py:35-45` (blueprint registration order keeps both surfaces active).
- Likely owner: backend
- Fix wave: Wave 2
